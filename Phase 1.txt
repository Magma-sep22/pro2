Phase 1: การกำหนดความต้องการ (Requirements Definition)
ปล่อยเรื่องโค้ดไปก่อน แล้วตั้งคำถามกับระบบ

Mission Requirements: ระบบนี้สร้างมาเพื่ออะไร? (เช่น รับ Telemetry, สั่งการ, บันทึก Log เพื่อวิเคราะห์)

Performance Requirements: ต้องการอัตรารีเฟรชเท่าไหร่? โมเดล 3D ต้องอัปเดตกี่ครั้งต่อวินาทีถึงจะไม่กระตุก? ความคลาดเคลื่อนของข้อมูลที่ยอมรับได้คือเท่าไหร่?

Phase 2: การออกแบบระบบและโปรโตคอล (System & Interface Design)
จุดที่เราควรโฟกัสเป็นอันดับแรกในตอนนี้

Hardware Constraints: ยอมรับข้อจำกัดของ NRF24L01 (32 ไบต์/แพ็กเก็ต) และพลังประมวลผลของ RP2040

Data Architecture: ออกแบบโครงสร้าง "Data Packet" ให้ชัดเจน (เช่น ใช้ Binary Struct, กำหนด Sync Word, ใส่ CRC16 เพื่อตรวจสอบความถูกต้อง)

Priority Scheduling: จัดลำดับความสำคัญว่าข้อมูลใดต้องส่งถี่ (เช่น IMU) ข้อมูลใดส่งช้าได้ (เช่น อุณหภูมิ, ฝุ่น)

Phase 3: การลงมือสร้าง (Implementation)
เมื่อแบบแปลนใน Phase 2 ชัดเจน การเขียนโค้ดจะตรงจุดและไม่สะเปะสะปะ

ฝั่งดาวเทียม (OBC): เขียน C++ ตามโครงสร้าง Struct ที่ออกแบบไว้, เปิดใช้งานฮาร์ดแวร์ฟิลเตอร์ (DLPF) เพื่อกรองสัญญาณรบกวน

ฝั่งภาคพื้น (Ground Station): เขียน Node.js ให้รับ Byte Buffer, ตรวจสอบ CRC ก่อนแพร่ข้อมูลลง Socket.io และเขียนระบบบันทึกลงไฟล์ CSV (Data Logging) ทันทีที่รับข้อมูล

Phase 4: การทดสอบระดับย่อยและการรวมระบบ (Unit Testing & Integration)
ทำทีละส่วนในสภาพแวดล้อมที่ควบคุมได้

Sensor Calibration: ปัญหา Gyroscope ที่เราเจอตกอยู่ในขั้นตอนนี้ ต้องเซ็ตอัปบนโต๊ะที่นิ่งสนิทและหาค่า Zero-Rate Offset ก่อน

Link Test: ส่งข้อมูลจำลอง (Dummy Data) ข้าม NRF24L01 เพื่อดูว่า Node.js ถอดรหัส CRC ถูกต้องและไม่ค้างเมื่อเจาะจงส่งข้อมูลวิบัติ (Corrupted Data) เข้าไป

Phase 1: OBC Firmware Fix (เน้นที่ดาวเทียม)
เป้าหมาย: กำจัดอาการ Gyro Drift และทำให้ข้อมูลนิ่งที่สุดก่อนส่งออกอากาศ

Revert to Old Code Logic: ใช้โครงสร้างการ init จากโค้ดเวอร์ชันเก่า คือต้องมีการสั่ง IMU.init(calib, IMU_ADDRESS); ซ้ำอีกครั้ง ทันทีหลังจากฟังก์ชัน calibrateAccelGyro ทำงานเสร็จ เพื่อให้ Bias ถูกนำไปหักล้างจริง

Activate DLPF (Hardware Filter): เพิ่มคำสั่ง IMU.setInternalFilter(20); ใน setup() เพื่อตัด Noise ความถี่สูง (กราฟฟันปลา) ออกตั้งแต่ต้นทาง

Buffer Safety: เปลี่ยนการใช้ sprintf ทั้งหมดเป็น snprintf(packet, PACKET_SIZE, ...) เพื่อจำกัดความยาวสตริงไม่ให้เกิน 32 ไบต์ ป้องกัน Pi Pico แครช

Physical Calibration: ทำการ Calibrate บนพื้นผิวที่เรียบและนิ่งสนิท ห้ามสัมผัสบอร์ดเด็ดขาดในช่วงที่ Buzzer ดังเตือน

📡 Phase 2: Data Link & Magnetometer Check (ตรวจสอบการเชื่อมต่อ)
เป้าหมาย: มั่นใจว่าข้อมูลมาครบทุกเซ็นเซอร์และส่งถึงผู้รับ

I2C Scan: รันฟังก์ชัน scani2c() เพื่อตรวจสอบว่าระบบมองเห็น Magnetometer (Address 0x0C) หรือไม่ หากไม่เห็น ให้ตรวจสอบการต่อสายหรือคำสั่ง Bypass ในโค้ด

Address Sync: ตรวจสอบให้มั่นใจว่าตัวแปร address ในโค้ดดาวเทียมและดองเกิลรับสัญญาณเป็นค่าเดียวกัน (เช่น EFEFE)

Serial Monitoring: ตรวจสอบข้อมูลดิบผ่าน Serial Monitor ของตัวรับสัญญาณก่อนดูที่แดชบอร์ด เพื่อยืนยันว่าค่า Gyro เข้าใกล้ 0.00 เมื่อวางนิ่ง

💻 Phase 3: Dashboard & Stability (คงสภาพแดชบอร์ด)
เป้าหมาย: แสดงผลโดยไม่ต้องรื้อระบบเดิม

Maintain Dashboard Logic: ไม่ต้องแก้ไขโค้ด JavaScript/HTML หากลำดับตัวแปรในแพ็กเก็ต #I และ #E ตรงกับของเดิม

Data Verification: ตรวจเช็คตำแหน่ง Index ของตัวแปรหลังจากส่งข้อมูลเวอร์ชันใหม่ขึ้นไป ว่าตัวเลขแสดงผลถูกช่องหรือไม่

Logging Check: ตรวจสอบการบันทึกไฟล์ CSV ว่าข้อมูลที่นิ่งแล้วถูกบันทึกไว้อย่างถูกต้องเพื่อใช้ทำ Report

🚀 Phase 4: Future Scaling (การขยายผลในอนาคต)
เป้าหมาย: เตรียมความพร้อมสำหรับดาวเทียมเกรดอุตสาหกรรม

Binary Protocol: เมื่อระบบ String เสถียรแล้ว จึงค่อยพิจารณาเปลี่ยนเป็น Binary Struct + CRC16 เพื่อเพิ่มประสิทธิภาพแบนด์วิดท์ (ถ้าจำเป็น)

Advanced Sensors: หากต้องการสเกลไปสู่ดาวเทียมจริง พิจารณาใช้เซ็นเซอร์เกรด Tactical เช่น BNO055 หรือ ICM-20948 ที่มีระบบจัดการ Noise ดีกว่า

สรุปสั้นๆ: โฟกัสที่ Phase 1 ให้จบก่อน ถ้า Gyro นิ่งและไม่ Drift ปัญหาที่เหลือในแดชบอร์ดจะหายไปเองเกือบทั้งหมด
